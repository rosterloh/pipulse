# PiPulse — Mousefood UI + Simulator Support Plan

## Goal

Replace the current raw `embedded-graphics` text drawing with a [**mousefood**](https://github.com/ratatui/mousefood) / Ratatui UI, and add an `embedded-graphics-simulator` path so UI changes can be tested on a development machine without physical hardware.

---

## What Is Mousefood?

`mousefood` is a `no_std`-compatible Ratatui backend that translates Ratatui's rendering commands into `embedded-graphics` draw calls. Any display that implements `embedded_graphics::draw_target::DrawTarget` — including the simulator — can be used as the target. This means the exact same UI code runs on the Pi's ST7789 screen and on your laptop in an SDL2 window.

---

## Architecture Overview

```
┌──────────────────────────────────────────────────────────┐
│                    ui.rs  (shared)                       │
│   build Ratatui layout, widgets, render into Terminal    │
└───────────────────┬──────────────────────────────────────┘
                    │ Terminal<EmbeddedBackend<D>>
         ┌──────────┴──────────┐
         │                     │
  feature = "sim"        feature = "hw" (default)
         │                     │
SimulatorDisplay<Rgb565>  mipidsi ST7789 display
  (SDL2 window)            (SPI + GPIO via linux-embedded-hal)
```

Both branches produce a type that implements `DrawTarget<Color = Rgb565>`. `mousefood`'s `EmbeddedBackend` wraps either one transparently.

## Step 3 — `metrics.rs`

Move the two metric functions verbatim from `main.rs`:

```rust
// src/metrics.rs
use std::{fs, net::IpAddr};
use get_if_addrs::get_if_addrs;

pub fn get_ipv4() -> String {
    match get_if_addrs() {
        Ok(addrs) => addrs
            .into_iter()
            .filter(|i| !i.is_loopback())
            .filter_map(|i| match i.ip() {
                IpAddr::V4(v4) => Some((i.name, v4)),
                _ => None,
            })
            .find(|(name, _)| {
                !name.starts_with("docker")
                    && !name.starts_with("veth")
                    && !name.starts_with("lo")
            })
            .map(|(_, ip)| ip.to_string())
            .unwrap_or_else(|| "no ip".into()),
        Err(_) => "no ip".into(),
    }
}

pub fn get_loadavg() -> String {
    fs::read_to_string("/proc/loadavg")
        .ok()
        .and_then(|s| {
            let mut p = s.split_whitespace();
            let a = p.next()?.to_owned();
            let b = p.next()?.to_owned();
            let c = p.next()?.to_owned();
            Some(format!("{a} {b} {c}"))
        })
        .unwrap_or_else(|| "? ? ?".into())
}
```

## Step 4 — `ui.rs` — Ratatui Layout

This file is **100% display-hardware-agnostic**. It takes a mutable reference to any `Terminal<EmbeddedBackend<D>>` and renders the UI.

```rust
// src/ui.rs
use mousefood::prelude::*;
use ratatui::{
    layout::{Constraint, Direction, Layout},
    style::{Color, Style},
    widgets::{Block, Borders, Paragraph},
    Terminal,
};

pub struct AppState {
    pub ip: String,
    pub load: String,
}

/// Render one frame. `terminal` is generic over the display backend.
pub fn render<D>(terminal: &mut Terminal<EmbeddedBackend<D>>, state: &AppState)
where
    D: embedded_graphics::draw_target::DrawTarget<Color = embedded_graphics::pixelcolor::Rgb565>,
    D::Error: core::fmt::Debug,
{
    terminal
        .draw(|frame| {
            // Split screen vertically: header (fixed 3 rows) + body
            let chunks = Layout::default()
                .direction(Direction::Vertical)
                .constraints([
                    Constraint::Length(3), // title bar
                    Constraint::Min(0),    // metrics area
                ])
                .split(frame.area());

            // ── Header ───────────────────────────────────────
            let title = Paragraph::new("PiPulse")
                .block(Block::default().borders(Borders::BOTTOM))
                .style(Style::default().fg(Color::Cyan));
            frame.render_widget(title, chunks[0]);

            // ── Metrics panel ─────────────────────────────────
            // Split body into two horizontal rows
            let metrics = Layout::default()
                .direction(Direction::Vertical)
                .constraints([Constraint::Length(3), Constraint::Length(3)])
                .split(chunks[1]);

            let ip_widget = Paragraph::new(state.ip.as_str())
                .block(Block::default().title("IP").borders(Borders::ALL))
                .style(Style::default().fg(Color::White));
            frame.render_widget(ip_widget, metrics[0]);

            let load_widget = Paragraph::new(state.load.as_str())
                .block(Block::default().title("Load").borders(Borders::ALL))
                .style(Style::default().fg(Color::Yellow));
            frame.render_widget(load_widget, metrics[1]);
        })
        .unwrap();
}
```

**Adding new widgets later is purely additive** — split the layout further, add a `Paragraph`, `Gauge`, `Sparkline`, etc. No display-layer code needs touching.

## Step 5 — `display/hw.rs` — Hardware Backend

This version uses `linux-embedded-hal` exclusively. Notable differences from the original:

- `SpidevDevice` opens `/dev/spidev0.0` directly and already implements `embedded_hal::spi::SpiDevice`, so `ExclusiveDevice` and `NoCs` are no longer needed.
- GPIO pins come from `gpio_cdev` (the Linux character device GPIO interface), accessed via `linux_embedded_hal::CdevPin`.
- Backlight is set to full brightness via a GPIO high. `linux-embedded-hal` does not expose hardware PWM — see the note below.

```rust
// src/display/hw.rs
//! Compiled only when the `hw` feature is active (the default).

use embedded_graphics::pixelcolor::Rgb565;
use linux_embedded_hal::{
    gpio_cdev::{Chip, LineRequestFlags},
    spidev::{SpiModeFlags, SpidevOptions},
    CdevPin, Delay, SpidevDevice,
};
use mipidsi::{interface::SpiInterface, models::ST7789, options::ColorInversion, Builder};

use crate::AppError;

const SPI_DC: u32   = 25;
const BACKLIGHT: u32 = 22;
pub const W: u16    = 240;
pub const H: u16    = 240;

/// Initialise GPIO, SPI, and the ST7789 display.
pub fn init() -> Result<impl embedded_graphics::draw_target::DrawTarget<Color = Rgb565>, AppError> {
    // ── GPIO (character device interface) ────────────────────
    let mut chip = Chip::new("/dev/gpiochip0").map_err(|_| AppError::Gpio)?;

    // DC pin (Data/Command select for the display)
    let dc = CdevPin::new(
        chip.get_line(SPI_DC)
            .map_err(|_| AppError::Gpio)?
            .request(LineRequestFlags::OUTPUT, 0, "pipulse-dc")
            .map_err(|_| AppError::Gpio)?,
    )
    .map_err(|_| AppError::Gpio)?;

    // Backlight — set high (full brightness)
    // Hardware PWM is not available via linux-embedded-hal.
    // To add dimming, use the `sysfs_pwm` crate targeting /sys/class/pwm/pwmchip0.
    chip.get_line(BACKLIGHT)
        .map_err(|_| AppError::Gpio)?
        .request(LineRequestFlags::OUTPUT, 1, "pipulse-bl")
        .map_err(|_| AppError::Gpio)?;

    // ── SPI ──────────────────────────────────────────────────
    // SpidevDevice already implements embedded_hal::spi::SpiDevice,
    // so ExclusiveDevice and the NoCs workaround are no longer needed.
    let mut spi = SpidevDevice::open("/dev/spidev0.0").map_err(AppError::Io)?;
    spi.configure(
        &SpidevOptions::new()
            .bits_per_word(8)
            .max_speed_hz(60_000_000)
            .mode(SpiModeFlags::SPI_MODE_0)
            .build(),
    )
    .map_err(AppError::Io)?;

    // ── Display ───────────────────────────────────────────────
    let buffer: &'static mut [u8; 512] = Box::leak(Box::new([0u8; 512]));
    let di = SpiInterface::new(spi, dc, buffer);
    let mut delay = Delay;

    Builder::new(ST7789, di)
        .display_size(W, H)
        .invert_colors(ColorInversion::Inverted)
        .init(&mut delay)
        .map_err(|_| AppError::Display)
}
```

## Step 6 — `display/sim.rs` — Simulator Backend

```rust
// src/display/sim.rs
//! Compiled only when the `sim` feature is active.

use embedded_graphics::pixelcolor::Rgb565;
use embedded_graphics_simulator::{
    OutputSettingsBuilder, SimulatorDisplay, SimulatorEvent, Window,
};

pub const W: u32 = 240;
pub const H: u32 = 240;

pub struct SimSetup {
    pub display: SimulatorDisplay<Rgb565>,
    pub window:  Window,
}

pub fn init() -> SimSetup {
    let display = SimulatorDisplay::<Rgb565>::new(
        embedded_graphics::geometry::Size::new(W, H),
    );

    let output_settings = OutputSettingsBuilder::new()
        .scale(3)          // 3× zoom so 240×240 fills a reasonable desktop window
        .pixel_spacing(1)  // tiny gap between pixels for that hardware look
        .build();

    let window = Window::new("PiPulse — Simulator", &output_settings);

    SimSetup { display, window }
}

/// Call after every draw. Pumps the SDL2 event loop; returns false when
/// the window is closed.
pub fn flush_and_poll(setup: &mut SimSetup) -> bool {
    setup.window.update(&setup.display);

    for event in setup.window.events() {
        if event == SimulatorEvent::Quit {
            return false;
        }
    }
    true
}
```

## Step 8 — `main.rs` — Wiring It All Together

`AppError` no longer references `rppal`. The SPI error variant becomes `Io(std::io::Error)` since `SpidevDevice` uses standard I/O errors.

```rust
// src/main.rs
mod display;
mod metrics;
mod ui;

use std::time::Duration;
use mousefood::prelude::*;
use thiserror::Error;

#[derive(Debug, Error)]
pub enum AppError {
    #[error("I/O error: {0}")]
    Io(#[from] std::io::Error),
    #[error("GPIO error")]
    Gpio,
    #[error("Display error")]
    Display,
}

fn main() -> Result<(), AppError> {
    #[cfg(all(feature = "hw", not(feature = "sim")))]
    run_hw()?;

    #[cfg(feature = "sim")]
    run_sim();

    Ok(())
}

// ── Hardware path ────────────────────────────────────────────────────────────

#[cfg(all(feature = "hw", not(feature = "sim")))]
fn run_hw() -> Result<(), AppError> {
    let display = display::hw::init()?;

    let config = EmbeddedBackendConfig::default();
    let backend = EmbeddedBackend::new(display, config);
    let mut terminal = Terminal::new(backend).unwrap();

    loop {
        let state = ui::AppState {
            ip:   metrics::get_ipv4(),
            load: metrics::get_loadavg(),
        };
        ui::render(&mut terminal, &state);
        std::thread::sleep(Duration::from_secs(1));
    }
}

// ── Simulator path ───────────────────────────────────────────────────────────

#[cfg(feature = "sim")]
fn run_sim() {
    let mut setup = display::sim::init();

    let config = EmbeddedBackendConfig::default();
    let backend = EmbeddedBackend::new(&mut setup.display, config);
    let mut terminal = Terminal::new(backend).unwrap();

    loop {
        let state = ui::AppState {
            ip:   metrics::get_ipv4(),
            load: metrics::get_loadavg(),
        };
        ui::render(&mut terminal, &state);

        if !display::sim::flush_and_poll(&mut setup) {
            break; // window closed
        }

        std::thread::sleep(Duration::from_millis(100)); // 10 Hz in sim
    }
}
```

## Step 9 — Remove `rppal`, Use `linux-embedded-hal`

The original codebase used `rppal` for GPIO and SPI. This step removes it and replaces it with `linux-embedded-hal`, which is already in the project for `Delay`. This makes the hardware abstraction layer consistent and removes a dependency.

### What changes

| Area | Before (`rppal`) | After (`linux-embedded-hal`) |
|---|---|---|
| SPI bus | `rppal::spi::Spi` → wrapped in `ExclusiveDevice<Spi, NoCs>` | `SpidevDevice::open("/dev/spidev0.0")` — already a `SpiDevice`, no wrapper needed |
| DC pin | `rppal::gpio::OutputPin` | `linux_embedded_hal::CdevPin` via `/dev/gpiochip0` |
| Backlight | `rppal` hardware PWM (`set_pwm_frequency`) | GPIO line set high at request time (full brightness) |
| Buttons | `rppal` input pin with pull-up | `CdevPin` with `INPUT` flag (still unused, wired for future use) |
| Error type | `rppal::spi::Error` | `std::io::Error` |
| `NoCs` struct | Required (rppal `SpiBus` needs a CS pin wrapper) | **Deleted** — `SpidevDevice` implements `SpiDevice` directly |
| `embedded-hal-bus` | Required for `ExclusiveDevice` | **Removed** from `Cargo.toml` |

### GPIO access: character device vs sysfs

`linux-embedded-hal` 0.4 uses the **character device GPIO interface** (`/dev/gpiochip0`) via the `gpio_cdev` sub-crate. This is the modern Linux GPIO API (available since kernel 4.8) and is preferred over the deprecated sysfs interface (`/sys/class/gpio`). On Raspberry Pi OS the gpiochip device is available out of the box with no extra configuration.

### PWM / backlight trade-off

`rppal` offered direct hardware PWM via its `set_pwm_frequency` method. `linux-embedded-hal` does not expose a PWM API. The backlight is wired to GPIO 22 which is also the Pi's `pwm0` hardware channel, so dimming is still achievable — it just requires a separate crate:

```toml
# Optional addition to Cargo.toml if hardware PWM dimming is desired later
sysfs_pwm = "0.2"
```

```rust
// Example usage with sysfs_pwm (optional future enhancement)
use sysfs_pwm::Pwm;
let pwm = Pwm::new(0, 0)?;       // chip 0, channel 0 → GPIO 22
pwm.export()?;
pwm.set_period_ns(20_000_000)?;  // 50 Hz
pwm.set_duty_cycle_ns(10_000_000)?; // 50%
pwm.enable(true)?;
```

For the initial implementation, full brightness (GPIO line value = 1) is sufficient and keeps the dependency surface minimal.

### `CAP_SYS_RAWIO` in the systemd unit

`rppal` required `CAP_SYS_RAWIO` to access `/dev/mem` for low-level hardware register manipulation. `linux-embedded-hal` uses the standard `/dev/spidev*` and `/dev/gpiochip*` device files instead. The capability can be replaced with group membership:

```ini
# scripts/pipulse.service — updated [Service] section
[Service]
Type=simple
ExecStart=/usr/local/bin/pipulse
Restart=on-failure
# No longer needs CAP_SYS_RAWIO — access via /dev/spidev0.0 and /dev/gpiochip0
# Add the service user to the `spi` and `gpio` groups instead:
#   sudo usermod -aG spi,gpio pipulse
User=pipulse
Group=pipulse
SupplementaryGroups=spi gpio
```

Or, if running as root (simplest for now), just remove the `AmbientCapabilities` line.

## Step 11 — Iterative UI Development Workflow

With this setup in place, the workflow for changing the UI is:

1. Edit `src/ui.rs` — adjust layout constraints, add/remove widgets, change colours.
2. `cargo run --no-default-features --features sim` — see the result instantly on your machine.
3. Once happy, `cargo build --release` and deploy to the Pi.

No hardware required for UI iteration.

## What Mousefood Gives Us (vs Current Approach)

| Concern | Before (raw embedded-graphics) | After (mousefood + Ratatui) |
|---|---|---|
| Text layout | Manual pixel arithmetic | Constraint-based `Layout` |
| Widgets | None — just `Text::new()` | Blocks, borders, gauges, sparklines, tables |
| Ghost pixels | Text redrawn over old text; stale chars possible | Ratatui does full-frame diffing; only changed cells are rewritten |
| Adding a new metric | Add another `Text::new()`, recalculate all Y positions | Add a `Constraint::Length(3)` row and a new `Paragraph` |
| Testing | Must have hardware | `cargo run --features sim` on any machine |
| Colour themes | Hardcoded `Rgb565::WHITE` | `Style::default().fg(Color::Cyan)`, full 256-colour palette via mousefood `ColorTheme` |

## Potential Extensions (Future Work)

- **Sparkline for load history**: accumulate a `VecDeque<f64>` of 1-minute load samples, render with `ratatui::widgets::Sparkline`.
- **Button A/B navigation**: current buttons are wired but unused. Ratatui's event model makes it straightforward to drive a tab/page system from GPIO interrupts.
- **More metrics panels**: CPU temperature (`/sys/class/thermal/thermal_zone0/temp`), memory (`/proc/meminfo`), uptime (`/proc/uptime`) — all follow the same `metrics.rs` + `ui.rs` pattern.
- **Custom colour theme**: `mousefood::ColorTheme` maps Ratatui's named colours to `Rgb565` values; a Pi-branded theme would take ~20 lines.
- **Hardware PWM dimming**: add `sysfs_pwm = "0.2"` and configure `/sys/class/pwm/pwmchip0` for the backlight (see Step 9).
- **Cross-compilation config**: add `.cargo/config.toml` with `target = "aarch64-unknown-linux-gnu"` and a linker entry so the simulator build stays native while the Pi build cross-compiles.
